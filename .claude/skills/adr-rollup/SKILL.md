# adr-rollup

카테고리 내에서 여러 ADR이 누적 진화(0001→0002→0003→...)한 경우, 이들을 하나의 "현재 상태" ADR로 통합(roll-up)한다.

## 목적

- 기능의 **현재 아키텍처 결정**을 한 파일에서 파악할 수 있게 한다
- 히스토리 ADR은 Superseded 상태로 보존하되, 새 통합 ADR이 source of truth가 된다
- 상세한 구현이나 중요한 아키텍처 결정은 반드시 유지한다

## Workflow

### 1. 대상 선정

사용자가 카테고리를 지정하면 해당 카테고리만, 아니면 전체 카테고리를 대상으로 한다.

`docs/adr/README.md`를 읽어 대상 카테고리의 ADR 목록을 파악한다.

### 2. 진화 체인 분석 — Roll-up 대상 결정

각 카테고리 내에서 **roll-up이 필요한 ADR 그룹**을 식별한다.

**Roll-up 대상**: 동일 기능/주제에 대해 순차적으로 개선·확장·대체한 ADR 체인
- 같은 기능의 초기 구현 → 개선 → 고도화
- 명시적 Superseded 관계
- 같은 기능의 다른 측면을 다루는 ADR들

**Roll-up 제외**: 독립적인 주제의 ADR은 roll-up하지 않는다
- 같은 카테고리지만 별개 기능 → 별개
- 이미 단독으로 완결된 ADR (진화 체인이 아닌 것)

**판단 기준**:
1. Related 섹션에서 같은 카테고리의 다른 ADR을 참조하는가?
2. 내용이 이전 ADR의 결정을 확장/수정/대체하는가?
3. 같은 엔티티/도메인 모델을 다루는가?

체인이 아닌 독립 ADR은 그대로 둔다.

### 3. Roll-up 전 전체 체인 읽기

Roll-up 대상으로 식별된 체인의 **모든 ADR을 읽는다**. 중요한 아키텍처 결정, 대안 비교, mermaid 다이어그램을 놓치지 않기 위해 반드시 전체를 읽어야 한다.

### 4. 통합 ADR 작성

체인의 마지막 번호 + 1로 새 ADR을 생성한다. (예: 0001~0004 체인 → 0005로 생성)

**통합 ADR 구조** — `docs/adr/README.md`의 ADR 템플릿을 따른다:

```markdown
# ADR {번호}: {카테고리} — 현재 아키텍처 (Roll-up)

Date: {오늘 날짜}

## Status

Accepted

## Context

{왜 이 기능/시스템이 필요한가 — 원래 문제와 진화 과정에서 드러난 문제를 통합}

## Decision

{현재의 아키텍처 결정을 서술한다. 과거 결정을 시간순으로 나열하지 않고, "현재 시스템이 이렇게 동작한다"를 쓴다}

### 대안 검토

{체인 전체에서 검토한 대안 중, 현재 결정을 이해하는 데 중요한 것만 모은다.
과거에 채택했다가 폐기한 접근도 "왜 폐기했는가"와 함께 기록한다}

## Consequences

### Positive
### Negative
### Risks

{현재 시스템 기준으로 작성. 이미 해결된 과거 리스크는 제거}

## Evolution History

| ADR | 주요 변경 | 현재 반영 여부 |
|-----|----------|--------------|
| 0001 | 초기 구현 | 부분 반영 (기본 구조 유지) |
| 0002 | XX 추가 | 완전 반영 |
| 0003 | YY 도입 | Superseded (0004로 대체) |
| 0004 | ZZ 통합 | 완전 반영 |

## Related

{현재 유효한 관련 ADR만 링크}
```

**작성 규칙** (docs/adr/README.md 작성 규칙 엄격 준수):

1. **현재 상태만 서술**: "~를 추가했다", "~로 변경했다" 같은 히스토리 서술을 쓰지 않는다. "~로 구성된다", "~를 사용한다"로 쓴다.
2. **중요한 아키텍처 결정은 반드시 유지**: 상태 전이, 행동 규칙, 엔티티 관계, 시스템 간 연동 방식, 비즈니스 로직 규칙
3. **Mermaid 다이어그램 보존**: 체인 내 다이어그램 중 현재 유효한 것은 반드시 포함. 여러 다이어그램이 있으면 현재 상태를 반영하도록 통합/수정한다
4. **대안 비교표 보존**: 현재 결정을 이해하는 데 중요한 대안 비교는 유지
5. **구현 세부사항 배제**: 파일 경로, 코드 스니펫, DB 필드별 스키마, 상수/튜닝값은 포함하지 않는다 (ADR README의 리트머스 테스트 적용)
6. **Error Handling 전략 유지**: graceful degradation, 폴백 패턴 등 아키텍처 수준의 에러 처리 전략은 유지
7. **Evolution History 섹션**: 체인의 각 ADR이 어떤 역할이었는지 한눈에 보이도록 테이블로 요약

### 5. 기존 ADR 상태 변경

체인 내 기존 ADR들의 Status를 변경한다:

```markdown
## Status

Superseded by [ADR {번호}]({파일명}.md) (Roll-up, {오늘 날짜})
```

**기존 ADR의 내용은 수정하지 않는다** — 히스토리 보존을 위해 Status만 변경한다.

이미 Superseded 또는 Deprecated인 ADR은 추가 변경하지 않는다 (이미 비활성).

### 6. README.md 업데이트

`docs/adr/README.md`의 카테고리별 목록을 업데이트한다:

- Roll-up에 통합된 기존 ADR 항목은 목록에서 제거한다
- 새 통합 ADR 항목을 추가하고 Key Decisions 요약을 작성

### 7. 사용자 확인

변경 사항을 요약하여 사용자에게 제시한다:

```
## ADR Roll-up 결과

### {카테고리}
- **통합 ADR**: 0005-{이름}.md (← 0001, 0002, 0003, 0004 통합)
- **핵심 결정**: {1-2문장 요약}
- **유지된 아키텍처 결정**: {목록}
- **제거된 내용**: {목록 — 이미 해결된 리스크, 폐기된 접근 등}

### 변경되지 않은 독립 ADR
- 0003-xxx.md, 0005-yyy.md, ...
```

사용자가 승인하면 파일을 저장한다. 승인 전까지는 저장하지 않는다.

### 8. 코드 동기화 검증

파일 저장 후, 통합 ADR의 핵심 아키텍처 결정이 실제 코드베이스와 일치하는지 검증한다.

adr-sync 스킬의 카테고리별 코드 경로 매핑을 참조하여 해당 카테고리의 코드를 grep한다.

**검증 절차:**

1. 통합 ADR에서 핵심 아키텍처 주장(architectural claims)을 추출한다
   - 엔티티 이름, 필드, 상태값
   - 시스템 간 연동 방식 (이벤트, API 호출 등)
   - 에러 처리 전략
   - 사용/미사용 여부가 명시된 기능
2. 각 주장에 대해 카테고리 코드 경로에서 grep으로 검증한다
3. **불일치 발견 시**: 코드가 source of truth이므로 통합 ADR을 코드에 맞게 수정한다. ADR README의 Key Decisions도 함께 갱신한다.
4. 검증 결과를 사용자에게 보고한다

```
## 코드 동기화 검증 결과

### 일치 확인
- ✅ DynamoDB 상태 머신 전이 패턴 확인
- ✅ S3 Vectors 유사도 검색 로직 존재 확인
- ...

### 불일치 발견 & 수정
- ⚠️ ADR에 "X를 사용한다"고 되어 있으나 코드에서는 "Y를 사용" → ADR 수정 완료
- ...
```

**검증 범위**: 아키텍처 수준의 결정만 검증한다. 구현 상수, 튜닝값, 파일 경로는 검증 대상이 아니다.

### 9. 기존 ADR 삭제

코드 동기화 검증까지 완료되면, 체인 내 기존 ADR 파일들을 **삭제**한다. 통합 ADR이 source of truth이므로 기존 파일을 유지할 필요가 없다. Git 히스토리에 원본이 보존되어 있으므로 컨텍스트 손실 없이 ADR 총 파일 수를 최소화할 수 있다.

- 이미 Superseded 상태로 변경된 기존 ADR 파일들을 `git rm`으로 삭제한다
- 통합 ADR의 Evolution History 섹션이 각 기존 ADR의 역할을 요약하고 있으므로 히스토리 컨텍스트는 유지된다
- README.md에서 해당 항목은 이미 제거되어 있으므로 추가 변경 불필요

## Notes

- Roll-up은 정보 손실이 아닌 **정보 압축**이다. 중요한 결정이 누락되면 안 된다.
- 의심스러우면 포함한다. 나중에 제거하는 것이 복구보다 쉽다.
- 카테고리 전체가 아닌 체인 단위로 roll-up한다. 같은 카테고리 내 독립 ADR은 건드리지 않는다.
- Roll-up 후 adr-sync를 실행하여 코드와의 동기화를 확인하는 것을 권장한다.
- 이 스킬은 docs/adr/README.md의 ADR 작성 가이드와 작성 규칙을 엄격히 따른다.
