## 8단계: 보고서 생성 (직접 수행)

보고서는 **사람이 읽고 의사결정하기 위한 문서**이다. 서술형으로 풀어 쓰고, 수치·타임스탬프·근거를 명시하되 기계 실행용 상세 명령은 포함하지 않는다 (그건 플레이북의 역할).

Google SRE 포스트모템과 Atlassian blameless postmortem 관행을 참조한 다음 구조로 한글 Markdown을 작성한다:

```markdown
## TL;DR

[한두 문장으로 장애·근본원인·조치·현재 상태를 요약. 매니저가 이 단락만 읽고 상황을 이해할 수 있어야 한다.]

## 인시던트 개요

| 항목 | 값 |
|------|-----|
| 알람 | [알람 이름] |
| 최초 감지 | [UTC 타임스탬프] |
| 복구 완료 | [UTC 타임스탬프 또는 "진행 중"] |
| 지속 시간 | [분 단위] |
| 심각도 | [low/medium/high/critical] |
| 영향 범위 | [single/service/regional] |
| 분석 확정 여부 | [확정 / 최유력 후보(신뢰도 X.XX)] |

## 영향 (Impact)

- **사용자 영향**: [영향받은 사용자 수·요청 수·비율. 불가 시 "측정 불가, 메트릭상 추정 N%"]
- **비즈니스 영향**: [수익/SLO 위반 여부]
- **시스템 영향**: [어떤 서비스·리소스가 어떻게 degrade되었는지]

## 근본 원인

[서술형 1-2단락. "무엇이 일어났는가" 보다 "왜 일어났는가"에 집중.]

### 5 Whys

증상 → 시스템·프로세스 결함까지 인과 사슬을 추적한다. **각 단계의 답은 수집된 증거로 뒷받침되어야 한다** (없으면 그 지점에서 사슬을 자른다). 사슬은 5단계까지가 권장이지만, 시스템 결함에 도달하면 더 일찍 멈춰도 좋다.

1. **왜** [증상]이 발생했는가? → [직접 원인]
2. **왜** [직접 원인]이 발생했는가? → [한 단계 아래]
3. **왜** ... → ...
4. **왜** ... → ...
5. **왜** ... → [근본 원인 — 시스템·프로세스·설계 결함]

> 신뢰도: **0.XX** — [확정 / 최유력 후보]

#### 5 Whys 작성 가드레일 (Toyota / AWS COE)

- **"휴먼 에러"·"운영자 실수"에서 멈추지 않는다.** 그건 종착점이 아니라 "왜 그게 가능했는가? (검증 부재? 권한 과다? 런북 결함?)" 라는 다음 단계를 요구하는 신호다. 시스템·프로세스 결함으로 풀어낸다.
- **단일 원인을 가정하지 않는다.** 다요인이 함께 작용한 경우, 주축 사슬은 가장 부하가 큰 요인으로 잡되 각 단계의 답에 기여한 공동 요인(co-factors)을 한 줄로 명시한다. 필요하면 별도 분기를 두 번째 사슬로 짧게 추가한다.
- **사실로만 내려간다.** 각 "왜?"의 답은 `## 뒷받침 증거` 표의 항목 또는 타임라인 이벤트와 연결되어야 한다. 추측이면 사슬을 거기서 자르고 *"이 시점부터는 증거 부족 — 추가 조사 필요"* 라고 명시한다.
- **Blameless.** 사람·팀을 비난하지 않는다. "X가 실수했다" 대신 "롤백 절차에 자동 검증이 없었다" 처럼 시스템 관점으로 표현한다.

## 뒷받침 증거

| # | 증거 | 출처 | 타임스탬프(UTC) |
|---|------|------|----------------|
| 1 | [구체적 수치·로그 스니펫. 예: `DatabaseConnections=215 (threshold=200)`] | CloudWatch AWS/RDS | 2026-04-29T12:03:00Z |
| 2 | [에러 메시지 한 줄] | CloudWatch Logs `/aws/ecs/...` | 2026-04-29T12:04:15Z |
| 3 | [배포·설정 변경 이벤트] | CloudTrail `UpdateService` | 2026-04-29T11:58:22Z |

## 타임라인

모든 시각은 **UTC**. 핵심 이벤트만 시간순으로.

- `T-15m` **12:00:00** — [정상 상태 기준점]
- `T+00`  **12:15:00** — **장애 시작**: [트리거 이벤트]
- `T+02`  **12:17:03** — 알람 발화 (CloudWatch)
- `T+02`  **12:17:30** — RCA 에이전트 분석 시작
- `T+07`  **12:22:45** — 근본원인 확정 (신뢰도 0.92)
- `T+08`  **12:23:10** — 임시 완화 조치 실행
- `T+09`  **12:24:00** — **복구 검증 완료**: 메트릭 정상화

2. └ **[자식 가설]** (신뢰도 0.75) → NEEDS_INVESTIGATION
3.   └ **[확정 가설]** (신뢰도 0.92) → **CONFIRMED**
```

### 작성 원칙 (보고서용)

- **Blameless**: 개인·팀을 특정해 비난하지 않는다. "deploy가 들어왔다" OK, "X가 실수했다" NO.
- **수치와 타임스탬프 필수**: 모든 증거에 구체 값과 UTC 타임스탬프. 없으면 "미측정"으로 명시.
- **서술형 > 단편적 bullet**: 근본원인·5Whys·교훈은 맥락을 담은 문장으로 작성.
- **미확정 시**: 신뢰도 < 0.9이면 근본 원인 섹션 첫 줄에 `> ⚠ 최유력 후보 (신뢰도 0.XX) — 미확정`을 명시.
- **명령어 금지**: MCP 호출·AWS CLI 등 기계 실행 명령은 포함하지 않는다. 그건 플레이북에 기록.

**반드시 `save_artifact("report.md", ...)` 로 저장한다. Python wrapper가 이 파일을 읽어서 S3에 업로드한다.**
