## 3-7단계: 검증 루프 (서브에이전트, 최대 3회 반복)

각 루프마다 Agent tool을 사용하여 **가설 검증 서브에이전트**를 스폰한다.

### 루프 진입 전: Accepted Review Gate (필수)

메인 에이전트는 **매 루프 진입 직전** 다음 게이트를 순수 로직으로 평가한다 (서브에이전트 스폰 전).

| 조건 | 액션 |
|------|------|
| 채택된(CONFIRMED) 가설 중 최고 신뢰도 ≥ 0.9 | **루프 즉시 종료** → 보고서 생성 |
| 0.8 ≤ 최고 신뢰도 < 0.9 | **비확장 모드**: 서브에이전트에게 `expansion_blocked=true` 전달. 증거 보강만 허용, 새 분기·재생성 금지 |
| 그 외 (채택 가설 없음 또는 최고 신뢰도 < 0.8) | 통상 흐름 |

**비확장 모드 연속 2회**에도 0.9 돌파 실패 시 → 루프 종료 → 보고서 생성.

**Accepted-유사 자동 기각**: 채택 가설이 있으면 PENDING/NEEDS_INVESTIGATION 가설 중 다음을 만족하는 것을 메인 에이전트가 REJECTED로 자동 전환한다:
- 같은 `category`
- `description` 토큰 오버랩(Jaccard 유사도) ≥ 0.6

자동 기각 시 reasoning: `"Review gate: 이미 채택된 {accepted_id}와 동일 원인 영역 — 자동 기각"`. 이 정리를 마친 뒤 서브에이전트를 스폰한다.

> 용어: "채택(ACCEPTED)"은 가설이 CONFIRMED 상태로 판정되었음을 가리키는 표기. JSON 스키마상 `status` 값은 하위 호환을 위해 `CONFIRMED`를 유지한다.

### 서브에이전트 프롬프트에 포함할 내용

- 현재 가설 목록 (모든 상태 포함)
- 스코핑 결과
- 알람 상세 정보
- 현재 루프 인덱스 (1-based)
- 기각된 가설 목록 (중복 분기 방지용)
- **채택된 가설 목록과 해당 신뢰도** (있을 시)
- **`expansion_blocked` 플래그** (비확장 모드 여부)

### 서브에이전트가 수행할 작업 (1회 루프)

1. **우선순위 결정**: PENDING/NEEDS_INVESTIGATION 가설을 정렬, 상위 3개 빔 선택
2. **증거 수집**: 빔 가설에 대해 CloudWatch/CloudTrail/GitHub MCP로 증거 수집
3. **가설 검증**: 신뢰도 평가
   - 신뢰도 ≥ 0.8 → CONFIRMED (채택)
   - 신뢰도 ≤ 0.3 → REJECTED (기각)
   - 0.3-0.8 → NEEDS_INVESTIGATION
4. **가설 분기**: NEEDS_INVESTIGATION 가설에서 2-3개 하위 가설 생성 (최대 깊이 3)
   - **`expansion_blocked=true`이면 이 단계를 건너뛴다** (분기·재생성 금지, 증거 보강만 수행)
   - 분기 시에도 기존 **채택된 가설과 같은 카테고리 + 유사 description**은 생성 금지 (중복 탐색 방지)
5. JSON 결과 반환

**메인 에이전트가 반환값을 받아 `save_artifact("validation-{N}.json", ...)` 으로 저장한다.**

### 루프 종료 판단 (메인 에이전트)

서브에이전트 반환값을 확인하여 **아래 순서로** 평가한다:

| 우선순위 | 조건 | 행동 |
|---------|------|------|
| 1 | **Accepted Review Gate**: 채택 가설 최고 신뢰도 ≥ 0.9 | 루프 즉시 종료 → 보고서 생성 |
| 2 | **Grace loops 소진**: 비확장 모드 2회 연속 + 여전히 < 0.9 | 루프 종료 → 현재 채택 가설로 보고서 |
| 3 | `confirmed` 가설 존재 (신뢰도 ≥ 0.9) | 루프 종료 → 보고서 생성 |
| 4 | 전체 경과 시간 > 8분 | 루프 종료 → 최선 결과로 보고서 |
| 5 | 루프 3회 완료 | 루프 종료 → 최선 결과로 보고서 |
| 6 | 모든 가설 기각 **+** 비확장 모드 | 재생성 스킵, 채택 가설 기반 보고서 |
| 7 | 모든 가설 기각 (비확장 모드 아님) | 재생성 (아래 참조) |
| 8 | 그 외 | 다음 루프 실행 (Gate 재평가부터 시작) |

### 전체 기각 시 재생성 (최대 2회)

모든 가설이 기각되면:
1. 가설 생성 서브에이전트를 다시 스폰한다
2. 프롬프트에 **기각된 가설 목록**을 포함한다
3. 서브에이전트는 기각 방향과 **다른 관점**에서 새 가설을 생성한다
4. 검증 루프를 재개한다

2회 재생성 후에도 확정 없으면 최고 신뢰도 가설을 "가장 유력한 후보"로 선정한다.

### 루프 종료 시 미검증 가설 처리 (필수)

**검증 루프 종료 후, PENDING 또는 NEEDS_INVESTIGATION 상태로 남은 가설은 최종 validation JSON의 `closed`에 포함한다.** 각 항목에 `hypothesis_id`, `confidence`, `reasoning`을 기재한다. reasoning에는 종료 사유(예: "시간 예산 소진", "확정된 근본원인 발견으로 추가 검증 불필요")를 명시한다. 확정된(CONFIRMED) 가설과 이미 기각된(REJECTED) 가설은 제외한다.

이 처리를 통해 세션 완료 시 모든 가설이 CONFIRMED, REJECTED, 또는 CLOSED 상태를 갖게 된다.
