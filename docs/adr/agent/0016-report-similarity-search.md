# ADR 0016: 보고서 기반 유사도 검색 — RCA 보고서 벡터 인덱싱으로 가설 생성 정확도 향상

Date: 2026-04-28

## Status

Accepted

## Context

이전 스코핑 단계에서는 S3 Vectors의 플레이북 인덱스를 검색하여 유사 장애 경험을 가설 생성에 반영했다(ADR 0001, ADR 0008). 그러나 플레이북은 "어떻게 대응하는가"(임시 조치, 영구 조치, 예방 조치)에 초점이 맞추어져 있어 가설 생성에 필요한 핵심 정보(증상 → 가설 경로 → 확정된 근본 원인)가 제한적이다.

RCA 보고서는 가설 생성에 더 적합한 정보를 담고 있다:
- `incident_summary`: 장애 증상 요약
- `root_cause`: 확정/추정 근본 원인
- `hypothesis_path`: 증상에서 근본 원인에 도달한 추론 경로
- `evidence_list`: 근본 원인을 뒷받침하는 증거

유사한 증상의 과거 보고서에서 "이 증상일 때 근본 원인은 X였고, Y 경로로 확인했다"라는 정보를 가설 생성에 직접 주입하면 초기 가설의 정확도가 크게 향상될 수 있다.

검토한 대안:

- **플레이북만 사용 (이전)**: 대응 절차 중심이라 가설 생성에 최적이 아님
- **보고서로 대체**: 가설 생성에 필요한 "증상 → 근본 원인" 정보를 가장 직접적으로 제공
- **보고서 + 플레이북 병행**: 두 인덱스를 모두 검색하면 정보가 풍부해지지만 스코핑 지연이 2배로 증가하고, 가설 생성 프롬프트가 비대해져 LLM 주의가 분산될 수 있음

## Decision

**스코핑 단계에서 플레이북 검색을 제거하고, RCA 보고서를 S3 Vectors에 인덱싱하여 유사 보고서 검색만으로 가설 생성에 활용한다.** 플레이북의 대응 절차(임시 조치/영구 조치)는 별도의 Remediation Agent(ADR 0012)에서 활용하며, 스코핑/가설 생성 경로에서는 참조하지 않는다.

### 핵심 결정사항

1. **동일 임베딩 템플릿 사용**: 보고서 인덱싱에 플레이북과 동일한 구조화 템플릿을 사용한다:
   ```
   장애유형: {root_cause} | 증상: {incident_summary} | 메트릭: {metric_name}
   ```
   동일 Cohere Embed V4 모델(1536차원)로 벡터화하며, 저장 시 `input_type=search_document`, 검색 시 `input_type=search_query`를 적용한다. 저장-검색 간 임베딩 공간 일관성과 플레이북-보고서 간 검색 품질 일관성을 보장한다.

2. **정성적(qualitative) 요약 전략**: 임베딩 필드(`incident_summary`, `root_cause`)에 구체적 수치(임계치, 퍼센트, 타임스탬프)를 포함하지 않는다. LLM 프롬프트에서 "abnormally high", "exceeds threshold", "sustained spike" 등 정성적 표현을 사용하도록 지시한다. 숫자가 다른 동일 패턴 장애(예: CPU 85% vs 92%)가 유사도 검색에서 매칭되도록 하기 위함이다. 보고서 생성 프롬프트(`REPORT_SYSTEM_PROMPT`)와 플레이북 생성 프롬프트(`PLAYBOOK_SYSTEM_PROMPT`)에 동일한 규칙을 적용한다.

3. **보고서 벡터 인덱싱 시점**: 보고서가 S3에 저장된 직후, 동일 파이프라인 내에서 벡터 인덱싱을 수행한다. 플레이북 인덱싱과 동일한 패턴(S3 저장 → 벡터 인덱싱)을 따른다.

4. **S3 Vectors 메타데이터**: `incident_summary`(80자), `root_cause`(80자), `hypothesis_path`(첫 번째 항목, 200자), `rca_id`를 메타데이터로 저장한다. 2048 bytes 제한에 맞춰 필드를 truncate한다.

5. **별도 인덱스**: 보고서 전용 인덱스(`S3_VECTOR_REPORT_INDEX`, 기본값 `report`)를 사용한다. 플레이북 인덱스와 분리하여 독립적으로 관리한다.

6. **가설 생성 프롬프트 주입**: 유사 보고서 검색 결과를 가설 생성 프롬프트에 `## Similar Past RCA Reports` 섹션으로 주입한다. 각 보고서의 `root_cause`와 `hypothesis_path`를 포함하여 LLM이 과거 경험을 바탕으로 가설을 세울 수 있도록 한다.

7. **검색 파라미터**: 유사도 임계치 0.7(`REPORT_SIMILARITY_THRESHOLD`), 상위 3건(`REPORT_TOP_K`). 플레이북 검색과 동일한 기본값을 사용한다.

8. **비차단 실행**: 보고서 검색 실패 시에도 파이프라인은 중단되지 않는다. 빈 결과로 진행하며, 기존 플레이북 검색 결과만으로도 가설 생성이 가능하다.

## Consequences

### Positive

- 과거 RCA의 "증상 → 근본 원인" 추론 경로를 직접 활용하여 가설 생성 정확도 향상
- 스코핑 시 벡터 검색 횟수가 기존(플레이북) 대비 동일하게 유지되어 지연 증가 없음
- 동일 임베딩 템플릿 + 정성적 요약으로 숫자가 다른 동일 패턴 장애 간 높은 유사도 매칭
- 가설 생성 프롬프트가 단일 참조 섹션(`## Similar Past RCA Reports`)으로 간결해져 LLM 주의 집중

### Negative

- 플레이북 벡터 인덱싱 기능은 유지되지만 스코핑/가설 생성 경로에서는 더 이상 조회되지 않음 (별도 Remediation Agent 등에서 활용 필요)
- 보고서가 축적되지 않은 초기 단계에서는 유사 검색 효용이 제한적이며, 검색 결과 없이 빈 컨텍스트로 가설 생성 진행
- 오판된 RCA 보고서의 근본 원인이 향후 가설 생성을 오도할 수 있음

### Risks

- `root_cause_confirmed=false`인 보고서의 추정 근본 원인이 높은 유사도로 매칭되면 잘못된 가설을 높은 우선순위로 생성할 수 있다. 검색 결과에 확정 여부(`confirmed`)를 표시하여 LLM이 가중치를 조절하도록 한다.
- 플레이북 검색 제거로 인해 과거 대응 절차 정보는 가설 생성 시점에 참조할 수 없다. 보고서의 `hypothesis_path`로 부분 보완되며, 실제 대응은 RCA 완료 후 플레이북 생성/복구 단계에서 별도 처리된다.
